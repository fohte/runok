## Why runok?

Claude Code's built-in permissions use simple glob matching. This works for basic cases, but breaks down quickly.

**Comments and compound commands cause unexpected prompts.** Even when `git log` is in your allow list, Claude adds a comment and you get prompted:

```
⏺ Bash(# check logs
      git log --oneline -5)

Command contains newlines that could separate multiple commands
```

runok strips comments and decomposes `&&` / `;` / `|` with tree-sitter-bash, evaluating each sub-command independently.

**Denied commands give no explanation.** The agent cannot learn why a command was blocked. With runok, deny rules include a message and a suggested fix:

```yaml
- deny: 'git push -f|--force *'
  message: 'Force push is not allowed.'
  fix_suggestion: 'git push --force-with-lease'
```

**Global flags break matching.** `git -C /path commit` does not match `git commit *`. runok supports [optional groups, alternation, and order-independent matching](/pattern-syntax/overview/):

```yaml
- allow: 'git [-C *] commit *'
# matches both "git commit" and "git -C /path commit"
```

[Read the full comparison &rarr;](/getting-started/why-runok/)

### How it works

Define allowed commands in `runok.yml`:

```yaml
rules:
  - allow: "git push|pull|fetch **"
  - allow: "npm install|test|run **"
  - deny: "rm -rf /"
```

Then use runok to check or execute commands:

```sh
$ runok check "git push origin main"
# ✅ allowed

$ runok check "rm -rf /"
# ❌ denied

$ runok exec "npm test"
# runs in a sandboxed environment
```
